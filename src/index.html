¿Qué es Spec Driven Development?

La metodología que pone las especificaciones en el centro del desarrollo

Spec Driven Development (SDD) es una metodología de desarrollo de software donde las especificaciones funcionales definen el comportamiento del sistema antes de escribir código. Cada feature comienza con una spec clara que describe qué debe hacer el sistema, bajo qué condiciones y con qué criterios de éxito. El código de implementación surge como consecuencia de la spec, no al revés. Esto elimina ambigüedades, reduce rework y asegura que el equipo construya exactamente lo que se necesita.

Principios Fundamentales

Specs Primero

Cada feature comienza con una especificación escrita que define comportamiento, entradas y salidas esperadas antes de cualquier implementación.

Claridad antes que Código

Las specs eliminan ambigüedades mediante lenguaje preciso, criterios de aceptación medibles y escenarios de prueba definidos por adelantado.

Iteración Controlada

Las specs se refinan en ciclos cortos de revisión antes de la implementación, evitando cambios costosos durante el desarrollo.

Colaboración Especificada

Roles humanos y de IA trabajan sobre artefactos compartidos (specs, planes, contratos) con responsabilidades claras en cada fase.

Calidad por Diseño

Los criterios de aceptación definen cuándo una feature está completa, asegurando que cada entrega cumpla con lo especificado.

Beneficios de SDD

Entregas Más Rápidas

Menos rework y retrabajo porque los requisitos están claros desde el inicio. El equipo implementa con confianza.

Menos Defectos

Las specs incluyen criterios de aceptación que se convierten en pruebas automatizadas, detectando desviaciones temprano.

Mejor Alineación del Equipo

Todos (desarrolladores, QA, producto) trabajan desde la misma fuente de verdad, reduciendo malentendidos.

Onboarding Acelerado

Los nuevos miembros del equipo pueden entender el proyecto leyendo las specs, sin necesidad de sesiones extensas de capacitación.

SDD vs Métodos Tradicionales

Aspecto Spec Driven Development Métodos Tradicionales
Requisitos Specs ejecutables con criterios de aceptación medibles antes de codificar Documentos vagos o historias de usuario sin criterios claros
Orden de trabajo Especificar → Planificar → Implementar → Verificar contra la spec Codificar primero, documentar después (si hay tiempo)
Colaboración Roles humanos e IA con responsabilidades definidas por fase Roles ambiguos, comunicación informal
Calidad Criterios de aceptación definidos antes; verificación automática Testing reactivo, bugs descubiertos tarde
Cambios Specs se versionan y actualizan; impacto evaluado antes de implementar Cambios ad-hoc durante desarrollo, scope creep frecuente

Ver detalle: Specify

Descripción

La fase Specify es donde todo comienza. Se escribe una especificación funcional que describe qué debe hacer el sistema, para quién y bajo qué condiciones. La spec incluye historias de usuario priorizadas, criterios de aceptación medibles y escenarios edge case. No se habla de implementación — solo de comportamiento deseado.

Artefactos de ejemplo

  • spec.md — Historias de usuario con Acceptance Scenarios (Given/When/Then)
  • checklists/requirements.md — Checklist de calidad de la spec

Preguntas frecuentes

¿Qué nivel de detalle debe tener la spec?

Suficiente para que cualquier desarrollador entienda qué construir sin ambigüedades, pero sin entrar en detalles de implementación.

¿Quién escribe la spec?

Típicamente el Spec Writer o Product Owner, con input de stakeholders y asistencia de IA.

Consejos

  • Empieza con el escenario más simple (happy path) y luego agrega edge cases
  • Cada criterio de aceptación debe ser medible y verificable
  • Usa el formato Given/When/Then para eliminar ambigüedad
Ver detalle: Clarify

Descripción

En Clarify se revisa la spec draft para identificar y resolver ambigüedades. Se hacen preguntas estructuradas sobre scope, comportamiento, datos y casos edge. El objetivo es que la spec quede lo suficientemente clara para planificar sin sorpresas.

Artefactos de ejemplo

  • spec.md (versión clarificada con respuestas incorporadas)
  • Lista de decisiones tomadas durante la clarificación

Preguntas frecuentes

¿Cuántas rondas de clarificación son normales?

Típicamente 1-2 rondas. Si hay más de 3, la spec probablemente necesita reescribirse desde una perspectiva más clara.

¿Puede la IA participar en la clarificación?

Sí, la IA puede generar preguntas de clarificación basadas en patrones comunes de ambigüedad.

Consejos

  • Prioriza preguntas por impacto: scope > seguridad > UX > detalles técnicos
  • Máximo 3-5 preguntas por ronda para mantener el foco
  • Documenta cada respuesta directamente en la spec
Ver detalle: Plan

Descripción

La fase Plan traduce la spec clarificada en un diseño técnico. Se definen la arquitectura, modelos de datos, contratos de interfaces, estructura de archivos y decisiones tecnológicas. El plan es el puente entre "qué" y "cómo".

Artefactos de ejemplo

  • plan.md — Contexto técnico, constitution check, estructura de proyecto
  • data-model.md — Entidades, campos, relaciones
  • contracts/ — Contratos de APIs o interfaces
  • research.md — Decisiones técnicas con alternativas consideradas

Preguntas frecuentes

¿El plan puede cambiar durante la implementación?

Sí, pero los cambios significativos deben documentarse y justificarse.

Consejos

  • Marca todo "NEEDS CLARIFICATION" que no esté decidido y resuélvelo antes de Tasks
  • El Constitution Check debe pasar antes de continuar
  • Mantén el plan lo más simple posible — justifica cada decisión de complejidad
Ver detalle: Tasks

Descripción

En Tasks se descompone el plan en una lista de tareas implementables. Cada tarea tiene un ID, descripción, archivo destino y se organiza por historia de usuario. Se identifican dependencias y oportunidades de paralelismo.

Artefactos de ejemplo

  • tasks.md — Lista completa de tareas con formato - [ ] T### [P?] [US#] Descripción
  • Dependency graph — Orden de ejecución y dependencias entre tareas

Preguntas frecuentes

¿Qué tan granular debe ser cada tarea?

Lo suficiente para que un LLM pueda completarla sin contexto adicional. Típicamente 1-4 horas de trabajo cada una.

Consejos

  • Organiza por user story para permitir entregas incrementales
  • Marca [P] solo si las tareas no comparten archivos ni dependencias
  • Incluye rutas de archivo exactas en cada descripción
Ver detalle: Implement

Descripción

La fase Implement es donde se ejecutan las tareas del plan. Se escribe código siguiendo las decisiones de arquitectura, se verifican los criterios de aceptación de la spec y se marca cada tarea como completada. El ciclo es: implementar → verificar → hacer commit → repetir.

Artefactos de ejemplo

  • Código fuente (src/)
  • tasks.md actualizado con tareas completadas - [x]
  • Commits con mensajes descriptivos

Preguntas frecuentes

¿Qué hacer si una tarea resulta más compleja de lo esperado?

Divídela en subtareas más pequeñas. Si cambia el enfoque, actualiza el plan y documenta por qué.

Consejos

  • Sigue el orden de tareas respeta las dependencias
  • Haz commit después de cada tarea o grupo lógico
  • Verifica contra la spec: ¿cumple los criterios de aceptación?
  • Trabaja en una rama feature y crea PR al terminar

Plantillas SDD

Constitución SDD

Una Constitución SDD define los aspectos fundamentales que todo proyecto debe contemplar para operar de manera predecible y mantenible. No es una lista de tecnologías específicas, sino un marco de gobierno que cubre stack tecnológico, arquitectura, workflow, calidad y documentación. Cada equipo adapta su constitución a su contexto.
